# Conductor Mesh Protocol — CC Integration

Claude Code instances can register on Anthropic's production conductor mesh (the same mesh Office Claudes use) and communicate with each other and with Office agents.

## Quick Start

### Option A: Full mesh-wrapped CC session (recommended)

```bash
claude-mesh                    # auto-named, auto-connected
claude-mesh -c                 # with continue flag (all claude flags pass through)
claude-mesh --no-mesh          # PTY wrapper only, no mesh
```

Launches CC inside a PTY wrapper with a bridge sidecar. Agent ID is auto-generated (`cc-{repo}-{hex6}`). Peers see the repo name in "Connected files". CC gets a startup orientation with available commands and online peers.

### Option B: Standalone bridge (for scripts/daemons)

```bash
npx tsx src/conductor-bridge.ts <agent_id> "<label>" [<color>] [<fileName>]
```

Example:
```bash
npx tsx src/conductor-bridge.ts cc-aboyeur "Aboyeur (CC)" "#7719AA" "aboyeur"
```

### Check status

```bash
cat /tmp/conductor-bridge/<agent_id>/status      # "connected" | "disconnected" | "reconnecting"
cat /tmp/conductor-bridge/<agent_id>/peers.json   # connected agents with labels and file names
cat /tmp/conductor-bridge/<agent_id>/events.jsonl # full protocol trace (all non-ping messages)
```

### Send a message

From inside a mesh-claude session, CC can use the `mesh` CLI:
```bash
mesh send <agent_id> "Hello"
mesh peers
mesh inbox
mesh status
```

Or write directly to the outbox:
```bash
echo '{"to": "cc-aboyeur", "message": "Hello"}' >> /tmp/conductor-bridge/<agent_id>/outbox.jsonl
```

### Read incoming messages

```bash
cat /tmp/conductor-bridge/<agent_id>/inbox.jsonl
```

Each line is: `{"ts": <unix_ts>, "from": "<sender_id>", "message": "<text>"}`

## Authentication

The bridge reads CC's own OAuth token. Location is platform-dependent:
- **Linux:** `~/.claude/.credentials.json` (flat JSON file)
- **macOS:** Keychain entry `Claude Code-credentials` (read via `security find-generic-password -s "Claude Code-credentials" -w`)

No additional auth setup needed — same token that authenticates your Claude Code session to Anthropic's servers.

## Agent Identity

Agent IDs are auto-generated by `mesh-claude.py` as `cc-{repo}-{hex6}` (e.g., `cc-aboyeur-a3f9e2`). This mirrors Office agents which use `{surface}-{hex6}` (e.g., `excel-d49606`). Both appear on the same mesh. However, **the ID format is free-form** — the `{surface}-{hex6}` pattern is a convention, not a protocol requirement (see Tested Behaviours below).

## Display Names and "Connected files"

Office apps show peers in a "Connected files" panel. The display name comes from **three sources with specific precedence**:

| Source | Schema field | What Office shows | When used |
|--------|-------------|-------------------|-----------|
| Registration | `schema.appName` → becomes `schema.label` on wire | Primary display | Always |
| Status broadcast | `payload.fileName` via `conductor_broadcast_status` | Updates display | After status event received |
| Registration | `schema.display.label` | NOT used for panel | Peer metadata only |

**For CC agents:** `mesh-claude.py` sets `appName` to `"{repo} ({suffix})"` (e.g., `"aboyeur (a3f9e2)"`) and broadcasts `fileName` with the same value via status event. This makes each CC session distinguishable in the Connected files panel.

**Note:** `display.label` (e.g., "Alice (CC)") is NOT what the Connected files panel shows — it uses `schema.label` (mapped from `appName`). This is non-obvious and was discovered by examining the raw `get_connected_agents` response from an Office Claude via the events.jsonl protocol trace.

## Wire Protocol Summary

| Direction | Type | Shape |
|-----------|------|-------|
| Client → Server | Register | `{"type": "register", "agentId": "...", "schema": {...}, "oauth_token": "..."}` |
| Server → Client | Connected | `{"type": "conductor_connected", "agentId": "...", "protocol_version": 2}` |
| Server → Client | Replay | `{"type": "conductor_event", "event_type": "connect\|status\|stream", "replay": true, ...}` |
| Server → Client | Replay done | `{"type": "conductor_replay_complete", "events_replayed": N}` |
| Client → Server | Send msg | `{"type": "conductor_send_message", "to": "...", "message": "...", "_agent_id": "..."}` |
| Server → Client | Recv msg | `{"type": "conductor_message", "from": "...", "message": "...", "_for_agent_id": "..."}` |
| Server → Client | Peer join | `{"type": "conductor_agent_online", "agentId": "...", "schema": {...}}` |
| Server → Client | Peer leave | `{"type": "conductor_agent_offline", "agentId": "..."}` |
| Client → Server | Status | `{"type": "conductor_broadcast_status", "payload": {...}, "_agent_id": "..."}` |
| Server → Client | Status event | `{"type": "conductor_event", "event_type": "status", "payload": {"fileName": "..."}}` |
| Client → Server | Keepalive | `{"type": "ping"}` |
| Server → Client | Keepalive | `{"type": "pong"}` |

All client messages in multiplexed mode require `_agent_id` **except pings** (pings must NOT include `_agent_id` from Node.js clients).

## Registration Schema

```json
{
  "type": "register",
  "agentId": "cc-aboyeur-a3f9e2",
  "schema": {
    "instructions": "I am aboyeur (CC), a Claude Code agent. Send me a task or message and I will respond.",
    "appName": "aboyeur (a3f9e2)",
    "version": "2",
    "interface": "Claude Code",
    "capabilities": {
      "receive_message": {},
      "file_sharing": {"accept": ["json", "txt", "md", "ts", "js", "py"]}
    },
    "display": {
      "label": "aboyeur (CC)",
      "color": "#7719AA"
    }
  },
  "oauth_token": "<from ~/.claude/.credentials.json>"
}
```

**`appName`** controls the display in Office's "Connected files" panel (the mesh maps it to `schema.label` on the wire). Set it to `"{repo} ({suffix})"` so each CC session is distinguishable.

**`capabilities.file_sharing.accept`** is visible to peers — they use it to decide what files to share. Office agents accept XML/SVG; CC agents accept source code formats.

## What's Proven (14 Mar 2026)

- CC's OAuth token (`sk-ant-oat01-*`) authenticates to the conductor mesh
- Registration succeeds, protocol version 2
- CC agents appear alongside Office agents (Excel, PowerPoint, Word)
- Bidirectional messaging works between CC agents
- Peer discovery works (online/offline events)
- 28+ buffered events replayed on connect (persisted mesh state)

## Critical: Node.js Required (15 Mar 2026)

**Python WebSocket libraries do NOT work for stable connections.** Tested `websockets` v16, `websocket-client`, and `curl_cffi` (Chrome TLS impersonation) — all get `"Stale connection (no pong)"` after ~60 seconds. The server closes with code 1001.

**Node.js `ws` library works perfectly.** Same token, same headers, same registration — pings get pongs, connection holds indefinitely. The TypeScript bridge at `src/conductor-bridge.ts` replaces the Python bridge.

The root cause is unknown but not TLS fingerprinting (curl_cffi disproved this), not headers (exact browser headers didn't help), not token scopes (CC and Office tokens both work from Node). The server appears to treat connections differently based on the WebSocket client implementation at a level below what we can control in Python.

**Ping format:** Send `{"type":"ping"}` WITHOUT `_agent_id`. From Node.js clients, the server responds with `{"type":"pong"}`. Including `_agent_id` on pings is harmless from Python but causes errors from Node.js. The `_agent_id` requirement applies to all other client messages (send_message, broadcast_status) but NOT pings.

## Tested Behaviours (15 Mar 2026)

Systematically tested during aby-dawugu characterisation. Each finding has direct evidence from controlled experiments on the production mesh.

### Offline message delivery: DROPPED

Messages sent to an agent that is not connected are **silently dropped**. No error to the sender, no queue on the server. When the target agent later connects, it does not receive the message.

**Implication:** Daemon must confirm a target is online (via peers.json or peer_online event) before sending. Fire-and-forget to potentially-offline agents will lose messages.

### Multiple agents per profile: NO LIMIT (tested to 5+)

5+ agents registered simultaneously from the same OAuth token / account UUID. All connected, all saw each other, all could exchange messages. No errors, no limit warnings.

**Implication:** Daemon can freely spawn multiple bridges (PM, workers, reflectors) without worrying about connection limits.

### Graceful deregistration: PARTIALLY WORKS

Sending SIGTERM to the bridge process triggers `ws.close()`, which causes the mesh server to emit `conductor_agent_offline` to CC peers. However, **Office "Connected files" panels retain stale entries** even after clean disconnection — observed via Excel screenshot showing 12 dead test agents still listed. Either Office caches aggressively, or `ws.close()` doesn't reliably propagate to all peer types.

Hard-killing the process (SIGKILL, or kill without signal handler) leaves stale peers everywhere. `mesh-claude.py`'s `os.killpg(SIGTERM)` pattern is better than nothing but not a complete solution.

**Implication:** Don't rely on peer lists being clean. Build tolerance for stale entries into any peer-discovery logic.

### Custom agentId format: FREE-FORM

Tested `my-custom-agent`, `aboyeur-daemon`, `a`, and a 55-character ID. All registered successfully. The `{surface}-{hex6}` pattern is an Office convention, not a protocol requirement.

**Implication:** Daemon agents can use meaningful names like `aboyeur-daemon`, `pm-aby-zehiwo-01`, `worker-aby-sanimu-01` directly as agent IDs.

### Rate limits: NONE at 50 msg burst

50 messages sent in a single burst from one agent to another. All 50 delivered in the same second, in order, zero errors. No throttling, no dropped messages, no warnings.

**Implication:** Message throughput is not a concern for coordination traffic. Even a chatty PM spawning multiple workers won't hit limits.

### Peer-list divergence across sessions

Two sessions connected at different times see **different peer lists**. Session A (connected earlier) did not see agents that connected and disconnected between A's connect time and B's connect time, while session B (connected later) did see them via event replay.

**Hypothesis:** Each session receives event replay from its own connect time. Peer lists are built client-side from events, not served as a snapshot. If an agent connects and disconnects between your replays, you may never know it existed.

**Also observed:** A session's peer list does not always include agents that connected after it. This suggests the mesh server may not always deliver `conductor_agent_online` events to all existing sessions — possibly a race condition or eventual consistency in event propagation.

**Additional finding:** When a mesh message arrives while the user is actively typing in CC's input box, the bracketed paste injection manifests as a `[Pasted Text]` marker in the input. It does not auto-submit, so it's non-destructive, but it does interrupt the typing flow.

### Re-registration with same agentId: WORKS

After a bridge crash, relaunching with the same agentId succeeds. The mesh accepts the new registration without complaint, even while stale entries from the old connection may still be visible to some peers. Messages route to the new connection immediately.

**Implication:** Sidecar auto-restart (aby-miviti) can reuse the same agentId without needing to generate a fresh one.

### "Agent not found" error on send: FATAL (bug)

Sending a message to a nonexistent agentId causes the mesh server to return a `conductor_error` with "Agent not found". The bridge emits this as an `error` event. If no error listener is registered (as in the CLI entry point), Node.js throws an unhandled error and the process crashes.

**Implication:** The bridge must handle `conductor_error` gracefully — log and continue, don't crash. Filed as part of aby-pamiwi (Mac side).

## What's Not Yet Proven

- Token refresh during active connection (token expires every ~24h) — requires long-duration test
- Long-duration connection stability (hours/days) — requires long-duration test
- File sharing through the conductor virtual filesystem
