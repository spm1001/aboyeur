The structured event log (events.jsonl) is the single most valuable debugging tool for mesh work. Adding it took 5 minutes; it immediately revealed that Office Claudes send `fileName` and `documentUrl` in their status broadcasts, that the mesh maps `appName` to `schema.label` on the wire (which is what the "Connected files" UI reads), and that `_agent_id` is required on broadcast_status messages. Without it, we spent multiple iterations guessing which field controlled the display name. With it, the answer was in the first trace. Future mesh characterisation work (aby-dawugu) should start by examining events.jsonl traces rather than theorising.

The npx→tsx process chain orphan problem is more insidious than it looks. When mesh-claude.py terminates the bridge sidecar, a simple `proc.terminate()` kills only the top `npx` process, leaving 3 child processes alive and connected to the mesh. These orphans register with the same agent_id, causing the mesh server to supersede the live connection every ~1 second. The fix is `preexec_fn=os.setsid` on spawn + `os.killpg()` on cleanup — standard Unix process group management, but easy to miss when the process chain is hidden behind npx.
