CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC blocks CC Channels API entirely. The env var prevents feature flag fetches on startup, which means the tengu that gates channels (likely tengu_hayate or tengu_harbor) never gets enabled. Symptoms: MCP tools (send_message, mesh_peers) work fine because they're standard MCP, but channel push notifications (inbound <channel> tags) silently fail. CC shows "--dangerously-load-development-channels ignored (server:conductor-channel) / Channels are not currently available" on startup. The var was set in two places: ~/.claude/settings.json (env block) and spawnAgent()'s HEADLESS_ENV. Removing it from both restored full bidirectional mesh via Channels MCP — confirmed with live round-trip to Office Excel agent. The var should never be set for mesh-enabled sessions. spawnAgent() now omits it entirely (the other DISABLE flags are fine, only NONESSENTIAL_TRAFFIC causes this). For interactive sessions, it was removed from settings.json. This is the kind of failure that looks like a code bug but is actually a configuration/feature-flag interaction — future Claudes debugging "channels not working" should check this first.
