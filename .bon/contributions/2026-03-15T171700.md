The daemon doesn't need its own Gmail API code. The original architecture-decisions.md (6 Mar) said "daemon polls Gmail directly, not via mise" because mise is an MCP server requiring a Claude session. But with mise moving to served mode (HTTP, not stdio), that objection dissolves. A served mise is a persistent process — the daemon can just call its endpoint. This means: (1) mise owns all Gmail auth in one place (one jeton token, one OAuth client for claude@planetmodha.com), (2) the daemon contains zero Gmail code — it just calls mise's "new mail since X?" endpoint every 60 seconds, (3) the one-shot Claude that wakes up to handle the email uses the same served mise instance for rich operations (fetch thread, draft reply). One auth surface, one Gmail integration, two consumers. This is a significant simplification from the original design and should be reflected in architecture-decisions.md when the integration spec (aby-becibi) is written.
