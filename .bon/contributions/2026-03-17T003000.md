The walkie-talkie session revealed something important about the architecture's transport layer. We assumed the mesh paste-block problem was about message length — it's actually about burst timing. Single messages of any size (tested to 8K) inject cleanly via PTY. The paste blocks happen when multiple queued messages fire in rapid succession (50ms gaps). The fix is trivial: one message per poll cycle.

More fundamentally, the walkie-talkie (file + PostToolUse hook) proved that PTY injection isn't the only delivery mechanism for CC sessions. Hooks deliver messages as additionalContext — no terminal involvement, no paste risk. This means Guéridon's -p mode sessions could receive mesh messages via stdin envelope (which Guéridon already uses for prompts), bypassing the entire PTY injection layer. The mesh transport and the message delivery are separable concerns.

The hookEventName field in CC's hook output JSON is required for additionalContext to surface — CC silently drops hook output without it. The field must match the triggering event, which you get by reading hook_event_name from CC's stdin JSON. This was discovered through live paired debugging (two Claudes using the mesh to debug the walkie, using the walkie to debug the mesh).

The deeper design question explored this session: what if every repo were an on-demand expert that any Claude could consult? The coordination doesn't need the Anthropic mesh — a file drop in `.inbox/` works. The mesh adds real-time cross-machine reach but shouldn't be a dependency. Guéridon is the natural spawn broker for consultations.
