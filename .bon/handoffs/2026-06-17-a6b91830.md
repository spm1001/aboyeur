## For Claudes to come

The Claude↔Claude mesh ("channels") works like this: a CC session loads an MCP server declared with the `claude/channel` capability via `--dangerously-load-development-channels server:NAME`; that server (aboyeur's `conductor-channel.ts`) holds a ws bridge to `bridge.claudeusercontent.com/v2/conductor/{account.uuid}`, so every session on one account auto-shares a room and peer messages arrive in-context as `<channel>` tags. Three facts learned the hard way, load-bearing for anyone extending this:

1. **MCP servers do NOT register via `settings.json`'s `mcpServers` — that key is inert.** Real registration is `.mcp.json` (project-scoped), `~/.claude.json` (user-scoped, via `claude mcp add`), or a **plugin**. Plugins make a server global by construction — `plugin:mise:mise` proves it — which is precisely why packaging conductor-channel as the *sonnette* plugin is the route to "available in every session, any cwd".
2. **The channel binds only at FRESH session birth.** A `claude -c` resume reuses the MCP-config snapshot from when the session was created, so it cannot be retrofitted onto the mesh. Whether a session *born with* the channel keeps inbound across a later resume is still an open question (aby-wodagu).
3. **On Team plan**, `policySettings.allowedChannelPlugins` in `/etc/claude-code/managed-settings.json` (machine-level, self-owned on hezza+Mac) drops both the dev flag and the approval dialog. Max is not supported (open Anthropic feature request).

Full frame, constraints, and scope cuts (Cowork/cross-user out of scope) live in `understanding.md`.
