## For Claudes to come

The restructure from factory to peer model wasn't just renaming — it was a genuine shift in how to think about multi-agent coordination. The factory model (daemon → aboyeur → PM → worker → reflector) was comfortable because it maps to human management hierarchies. But it produced three problems: (1) the PM layer was speculative complexity that blocked actual progress, (2) the hierarchy implied authority relationships that activate deference rather than honest exchange, and (3) it required building coordination infrastructure (structured verdicts, managed queues, eyesight filters) before the system could do anything useful.

The peer model inverts this: bons provide coordination (what needs doing), mesh provides communication (peer-to-peer via `<channel>` tags), and the daemon provides spawning (wake a Claude when something happens). No management layer. The complexity of coordinating work lives in the existing bon system, not in a new orchestrator. "Aboyeur" is a role any Claude can wear — triager, reviewer, worker — not a persistent entity at the top of a hierarchy.

Two technical insights that save significant debugging time: (1) CC spawns MCP servers twice in interactive mode due to a race between `prefetchAllMcpResources` (fire-and-forget in main.tsx) and `useManageMCPConnections` (React hook). The memoize cache is meant to deduplicate, but React effect re-runs can clear it. Any MCP server maintaining persistent external connections (WebSocket, database) should handle being spawned twice gracefully. (2) "Superseded by new connection" is the conductor mesh's close reason (code 1001) when another agent registers with the same agentId — yield, don't reconnect.
