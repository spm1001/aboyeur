## For Claudes to come

**The conductor mesh's settled character (measured, not assumed):** it is a *synchronous presence fabric* — a real-time party line for peers who are both alive — not a message bus. Three load-bearing consequences: (1) **No persistence.** A message to an absent peer is rejected synchronously (`Agent not found`), never buffered; the only "replay" is roster catch-up. So durable coordination MUST live in bons/files; the mesh carries only live-peer nudges. (2) **The peer list lies about the present.** Roster is eventually-consistent (a crashed peer shows "connected" for the 60–120s expiry window) while routing evicts instantly — reachability is a send + watching for the synchronous error, never `mesh_peers`. (3) **The flag/plugin is the membership boundary.** Loading the MCP server gives *outbound* tools; only being *born* with `--dangerously-load-development-channels` (or the future allowlisted sonnette plugin) makes inbound wire-arrivals surface as `<channel>` tags. A flagless session receives at transport but CC drops it before context — so "enmeshed" means born-with-flag, which is precisely why sonnette (plugin + managed-settings allowlist) is the vehicle for "every session born on the mesh."
